🔁 Репликация снапшотов

Автоматическая инкрементальная копия снапшотов на удалённый NAS или резервный сервер. Использует нативный механизм btrfs send / receive — передаются только изменения между снапшотами, что минимизирует трафик и сохраняет Btrfs-структуру.

🖥️ Исходный NAS
📂 Субволюм Btrfs
/.snapshots/…
btrfs send — потоковый дамп снапшота
→ SSH 🔒
🖥️ Удалённый хост
📂 Субволюм Btrfs
/backup/nas/…
btrfs receive — приём и применение дампа

🔒 SSH туннель · аутентификация по ключу · шифрование трафика

📦 Первая репликация
→ Полная передача всего снапшота
→ Размер = объём данных субволюма
→ Выполняется однократно
⚡ Последующие репликации
→ Только изменения между снапшотами (δ)
→ Минимальный трафик
→ По расписанию cron автоматически

Типичные сценарии

01
Резервный NAS в той же сети
Второй NAS в серверной — полная копия всех данных. При выходе основного из строя данные доступны немедленно без восстановления из облака.
02
Georedundancy / DR-сайт
Репликация через VPN/WireGuard на удалённую площадку. RPO = интервал расписания (например, 1 час). Защита от физического уничтожения сервера.
03
Удалённый офис
Синхронизация данных между двумя офисами. Каждый офис держит актуальную копию снапшотов коллег без ручного копирования.

Быстрый старт

1
На удалённом хосте создайте Btrfs-субволюм для приёма:
btrfs subvolume create /backup/nas
→
2
Скопируйте SSH-ключ NAS на удалённый хост:
ssh-copy-id user@remote-host
→
3
Нажмите + Добавить, заполните хост / пользователь / путь, нажмите Тест SSH
→
4
Нажмите ▶ Запустить для первой полной репликации. Далее — по расписанию автоматически.

📸 Снапшот — это не бэкап

Снапшот и бэкап решают разные задачи. Понимание разницы — основа правильной стратегии защиты данных.

📸 Снапшот Btrfs
✓ Создаётся мгновенно (секунды)
✓ Почти не занимает места сразу
✓ Откат в несколько кликов
✓ Хранится прямо на томе, быстрый доступ
✗ На том же диске — если диск умер, снапшоты тоже
✗ Не защищает от удаления файлов операционной системой
✗ Не защищает от физической катастрофы
💾 Бэкап (резервная копия)
✓ Хранится на другом устройстве
✓ Защищает от отказа оборудования
✓ Защищает от пожара, кражи (если offsite)
✗ Требует места на втором устройстве
✗ Создаётся дольше, восстановление — тоже
Правило 3-2-1: 3 копии данных, 2 разных носителя, 1 — вне помещения.
Снапшоты — часть «3 копий», но заменить бэкап на отдельном устройстве они не могут.
Правильная схема: Снапшоты (быстрый откат на месте) + Репликация на второй NAS или сервер (бэкап).

⚙️ Как работает снапшот внутри (CoW)

Btrfs использует Copy-on-Write: при создании снапшота новые данные не копируются — снапшот просто «указывает» на те же блоки данных, что и оригинал.

Субволюм
A.docx
1 ГБ
video.mp4
3 ГБ
backup.zip
500 МБ
Снапшот
(сразу после)
A.docx ↑
0 Б
video ↑
0 Б
backup ↑
0 Б
← Exclusive size = 0 Б. Только ссылки.
После
изменений
A.docx
(новая ver)
video ↑
0 Б
deleted.tmp
← Снапшот хранит старый A.docx. Новая версия записана рядом.
Размер снапшота в таблице = только уникальные блоки (Exclusive). Свежий снапшот без изменений = 0 Б. Чем дольше живёт снапшот и чем больше изменений — тем больше он «весит».
Из этого следует: удаление снапшота не освобождает место сразу «на всю папку» — Btrfs удаляет только блоки, уникальные для этого снапшота. Блоки, на которые ещё ссылается живой субволюм или другой снапшот — остаются.

Как выбрать значения: Думайте не о количестве снапшотов, а о временно́м охвате. Вопрос: «На сколько назад я должен уметь откатиться?»

🔁 Репликация — снапшоты на удалённый сервер

Репликация копирует снапшоты на второй NAS или сервер по SSH. Первая передача — полный снапшот. Каждая следующая — только изменения (инкрементально). Это и есть настоящий бэкап за пределами основного устройства.

🖥️ rusNAS (источник)
📁 /mnt/data/documents
📸 @2026-03-17 (снапшот)
📸 @2026-03-16 (предыдущий)
btrfs send -p @prev @new
→ передаёт только diff
SSH
🖥️ Удалённый сервер
📁 /backup/nas1/documents
📸 @2026-03-17 ← получен
📸 @2026-03-16 ← получен
btrfs receive /backup/nas1
← принимает diff, применяет

Как работает инкрементальность

День 1
(первый раз)
Передаётся полный снапшот: 1.7 ГБ → SSH → удалённый сервер
День 2
(следующий)
Изменилось 50 МБ → передаётся только diff 50 МБ. Предыдущий снапшот как «база».
День 3+
Аналогично: только новые изменения. Трафик → минимальный.
Что хранится на удалённом сервере? Полноценные read-only Btrfs-снапшоты. Восстановление: подключиться по SSH, примонтировать папку, скопировать файлы.

Настройка — пошагово

1
Подготовить SSH-ключ на NAS
Репликация работает без пароля — через SSH-ключ. Выполните на NAS от root:
ssh-keygen -t ed25519 -N "" -f /root/.ssh/id_rusnas_repl
2
Скопировать ключ на удалённый сервер
Заменить user@remote на вашего пользователя и адрес:
ssh-copy-id -i /root/.ssh/id_rusnas_repl.pub user@remote-host
3
Создать папку назначения на удалённом сервере
Папка должна находиться на Btrfs-разделе:
mkdir -p /backup/nas1/documents
4
Создать задачу репликации в UI
Вкладка «Репликация» → кнопка «+ Добавить». Выбрать субволюм, указать host/user/path, выбрать расписание.
5
Нажать «Проверить SSH» и «Запустить сейчас»
Первый запуск — полная передача снапшота. Следите за прогрессом в журнале. Последующие — только изменения.
Важно: на удалённом сервере папка назначения должна быть на Btrfs. btrfs receive не работает на ext4, xfs или NTFS.

✅ Чеклист: что настроить при первом запуске

1️⃣
Создать расписание для каждого субволюма
Вкладка «Расписание» → «+ Добавить». Минимум: daily каждую ночь.
2️⃣
Настроить Retention Policy
Без неё снапшоты будут накапливаться и займут всё место на диске.
3️⃣
Проверить, что снапшоты создаются
На следующий день откройте список — снапшоты должны появиться.
4️⃣
Сделать пробное восстановление
Создайте тестовый файл, создайте снапшот, удалите файл, восстановите через «Просмотр».
5️⃣
Настроить репликацию (если есть второй сервер)
Только так снапшоты станут полноценным бэкапом. SSH-ключ + задача репликации.
6️⃣
Заблокировать важные снапшоты
Например, снапшот «перед обновлением» — нажмите «Заблокировать». Retention не удалит его.
Золотое правило: снапшот — это откат за секунды. Репликация — это безопасность. Используйте оба.
