## ✅ ИСПРАВЛЕННЫЕ БАГИ (эта сессия)

### BUG-01 — Снапшоты: вкладка «Расписание» не обновлялась при смене субволюма
**Страница:** /rusnas/snapshots
**Статус:** ✅ Исправлено
**Описание:** Вкладка Расписание показывала данные предыдущего субволюма после переключения.
**Root cause:** `renderSchedules()` перебирал `schedulesData` вместо отфильтрованного по `currentSubvol` списка.
**Исправление:** Добавлена фильтрация `schedulesData.filter(s => s.subvol_path === currentSubvol)`.

---

### BUG-02 — Хранилище: кнопка «Изменить» SMB шары не открывала модал
**Страница:** /rusnas
**Статус:** ✅ Исправлено
**Описание:** Клик на «Изменить» в таблице SMB шар не открывал модал редактирования.
**Root cause:** `openEditShare()` использовал нестабильный Python inline-скрипт без `.fail()` обработчика; при любой ошибке spawn функция молча падала и модал не показывался.
**Исправление:** Переписан `openEditShare()` — теперь использует `grep -A20` bash-команду, вспомогательную функцию `_showEditModal()`, и `.fail()` с fallback-значениями.

---

### BUG-03 — Guard: `sudo rusnas-guard --reset-pin` — команда не найдена
**Страница:** /rusnas/guard (диалог ввода PIN)
**Статус:** ✅ Исправлено
**Описание:** Диалог ввода PIN показывал подсказку `sudo rusnas-guard --reset-pin`, но команда не работала на сервере.
**Root cause (1):** Симлинк `/usr/local/sbin/rusnas-guard` не существовал на VM.
**Root cause (2):** `install-guard.sh` содержал ошибку — `chmod 644 *.py` выполнялся ПОСЛЕ `chmod 755 guard.py`, сбрасывая бит исполнения. Также `/etc/rusnas-guard/` имел права 700 вместо 755.
**Root cause (3):** На VM был задеплоен старый `guard.py` без обработчика `--reset-pin`. Новый код был только локально.
**Исправление:** Создан симлинк вручную, исправлен порядок chmod в `install-guard.sh`, задеплоен обновлённый `guard.py` с логикой `--reset-pin`.

---

### BUG-04 — Хранилище: NFS раздел показывал «Ошибка:» при пустом /etc/exports
**Страница:** /rusnas
**Статус:** ✅ Исправлено
**Описание:** Секция NFS показывала красную ошибку вместо «Нет NFS экспортов» когда `/etc/exports` пустой.
**Root cause:** `grep` завершается с exit code 1 когда нет совпадений → `cockpit.spawn().fail()` срабатывал вместо `.done()`.
**Исправление:** Добавлен `|| true` в конец bash-команды + `{superuser: "require", err: "message"}` в spawn options.

---

### BUG-05 — Хранилище: SMB таблица показывала неверные пути шар (hardcoded)
**Страница:** /rusnas
**Статус:** ✅ Исправлено
**Описание:** Колонка «Путь» в таблице SMB шар всегда показывала `/mnt/data/shares/<name>` — захардкоженный путь, не совпадающий с реальным путём из smb.conf. Например, шара `documents` имела путь `/mnt/data/documents` в конфиге, но таблица показывала `/mnt/data/shares/documents`.
**Root cause:** `loadShares()` строил таблицу только по именам шар, подставляя дефолтный путь.
**Исправление:** `loadShares()` теперь парсит `testparm -s` через python3, получая реальные пути для каждой шары.

### BUG-06 — Снапшоты: «Закрыть доступ» случайно открывает модал «Метка»
**Страница:** /rusnas/snapshots → кнопка «Просмотр» → модал «Снапшот смонтирован»
**Серьёзность:** Средняя (UX нарушение, не ломает данные)
**Описание:** После нажатия «Закрыть доступ» (размонтирует SMB шару снапшота и закрывает модал) происходит непреднамеренное нажатие на кнопку «Метка» в строке таблицы позади модала — открывается модал «Изменить метку».
**Root cause:** Кнопка «Закрыть доступ» находится над оверлеем, но после закрытия модала click event «проваливается» сквозь overlay и попадает на кнопку в таблице. Оверлей исчезает до окончания обработки события.
**Как исправить:** В обработчике `browse-umount-btn` добавить `event.stopPropagation()`. Или: закрывать модал через `setTimeout(..., 50)` после unmount, давая оверлею время перехватить click.

## ⚠️ UX ПРОБЛЕМЫ И ЗАМЕЧАНИЯ

### Guard (/rusnas/guard)

**G-01: Кнопка «Выключить защиту» активна когда мониторинг уже выключен**
Красная кнопка «Выключить защиту» всегда видна, даже когда мониторинг неактивен. Это пугает неподготовленных пользователей. Кнопка должна быть disabled или скрыта когда `daemon_running = false`.
*Риск для пользователя:* нажмёт «Выключить» когда уже выключено → непонятное поведение.

**G-02: Колонка «IP ИСТОЧНИКА» показывает «unknown» для всех событий**
Файловые события от honeypot не имеют IP — это нормально, но колонка «IP ИСТОЧНИКА» создаёт ложное ощущение что данные потеряны. Лучше переименовать в «Субъект» или не показывать «unknown», а заменять на «—».
*Риск для пользователя:* непонимание что «unknown» нормально, или наоборот — игнорирование реальных IP при атаке.

**G-03: Колонка «Действие» показывает технические метки вида `logged + msg_ySysOmvdec`**
Идентификаторы лог-сообщений (`msg_*`) попадают в UI. Пользователь видит непонятный технический мусор. Нужно показывать только «Зарегистрировано» / «Заблокировано» / «Снапшот создан».
*Как исправить:* в `events.jsonl` хранить `action` как человекочитаемое значение, или фильтровать в JS при рендеринге.

**G-04: Кнопка «Снять все блокировки» отображается когда заблокированных IP = 0**
Красная кнопка в заголовке журнала событий всегда видна. Если нет заблокированных IP — кнопка должна быть скрыта или disabled. Пугает пользователей.

**G-05: «Базовый уровень: Обучение» — нет контекста**
Пользователь не знает что это значит, сколько длится «обучение» и что делать. Нужно добавить тултип или подсказку: «Guard изучает нормальную активность файловой системы в течение 24 часов».

**G-06: Много событий «Приманка» без объяснения — норма или атака?**
При первом открытии в журнале показываются десятки событий типа «Приманка» с датой вчерашнего дня. Непонятно — это нормальная работа или признак атаки? Нужен текст-пояснение: «Приманки — это защитные файлы Guard. Доступ к ним — сигнал тревоги. Если вы не выполняли резервное копирование — это подозрительная активность.»

**G-07: При POST-ATTACK состоянии при включении защиты появляются два подряд диалога PIN**
Если на сервере остался флаг `/etc/rusnas-guard/post_attack`, пользователь при нажатии «Включить защиту» получает сначала диалог «Включить защиту» → вводит PIN → сразу появляется новый диалог «Восстановить режим» → снова PIN. Это неинтуитивно.
*Как улучшить:* Предупреждать на странице баннером о post-attack состоянии ДО нажатия кнопки, чтобы пользователь понимал что произойдёт.

### Снапшоты (/rusnas/snapshots)

**S-01: Глобальный счётчик «30 снапшотов» не совпадает с суммой в сайдбаре (4)**
`storage-info` подсчитывает ВСЕ Btrfs subvols на системе включая тестовый пул `/mnt/btrfspool/`, а сайдбар показывает только управляемые субволюмы. Пользователь видит «30 снапшотов» но в списке только 4.
*Как исправить:* `storage-info` должен считать только субволюмы из БД расписаний + текущей области видимости. Или добавить пояснение «включая снапшоты вне управления».

**S-02: Размер «0 Б» для снапшотов — вводит в заблуждение**
Когда Btrfs qgroups отключены (типично для нового тома), `get_snap_size()` возвращает 0. Пользователь думает снапшоты пустые или что-то пошло не так. Нужно показывать «—» или «н/д» вместо «0 Б».

**S-03: Модал «Восстановить» не предупреждает об остановке сервисов**
При нажатии «Восстановить» появляется предупреждение «Текущее состояние будет заменено снапшотом». Но нет важного предупреждения: «Убедитесь что SMB/NFS шары на эту папку недоступны во время восстановления. Активные подключения могут привести к повреждению данных».
*Риск:* Пользователь восстановит снапшот пока другие пользователи работают с файлами.

**S-04: Модал «Расписание» — нет пояснения что значат параметры Retention**
Поля «Hourly (24 ч)», «Daily (14 д)» и т.д. непонятны без опыта работы с retention policy. Нужна краткая подсказка: «Оставлять 24 последних часовых снапшота, 14 ежедневных и т.д. Снапшоты удаляются автоматически по расписанию».

### Хранилище (/rusnas)

**H-01: Нет индикации статуса NFS сервиса**
В разделе NFS отображаются экспорты, но нет информации запущен ли `nfs-server`. Если сервис упал — пользователь не поймёт почему шары недоступны.
*Как улучшить:* Добавить бейдж статуса: `systemctl is-active nfs-server` → «Сервис: активен / остановлен».

**H-02: Нет индикации статуса Samba сервиса**
Аналогично NFS — нет статуса `smbd`. Шары показаны как «Активна» по статусу в smb.conf, но не по реальному состоянию сервиса.

**H-03: SMB — кнопка «Удалить» без подтверждения выглядит как немедленное действие**
В таблице SMB шар кнопка «Удалить» стоит рядом с «Изменить» без визуальной индикации что потребуется подтверждение. Пользователь может нажать случайно. (В коде есть `confirm()` — ок, но выглядит пугающе.)
*Рекомендация:* Добавить дополнительный отступ или `data-confirm` стиль к кнопке.

### BUG-13 — Dashboard: TypeError в dashboard.js:735 при каждой загрузке
**Статус:** ✅ Исправлено (dashboard.js)
**Описание:** `document.getElementById("card-snaps")` возвращал null — элемент с таким ID не существует в dashboard.html. Строки addEventListener для card navigation были избыточны (карточки уже имеют `onclick="cockpit.jump(...)"` в HTML).

**Fix:** удалён весь блок card navigation addEventListener (строки 731-735 в dashboard.js). Навигация через `onclick="cockpit.jump()"` в HTML работала корректно.

**Статус:** ✅ Исправлено (app.js:705)
**Root cause:** regex `/<Directory\s+(\S+)/m` захватывал `>` из тега `<Directory /path>`.
**Fix:** заменён на `/<Directory\s+([^>]+?)>/m`.

### SA-BUG-01 — Forecast data: 1 точка — нет прогноза
**Статус:** ⚠️ Ожидаемо (design decision)
**Описание:** После первого сканирования прогноза нет — только 1 точка истории. Прогноз появится после ≥2 сканирований.
**Action:** Добавлено объяснение в UI в карточке forecast (показывает "нет данных").

### SA-BUG-02 — DeprecationWarning в storage-collector.py:120
**Статус:** ⚠️ Minor warning, не влияет на работу
**Описание:** `re.split(r'\s*=\s*', line, 1)` передаёт maxsplit как positional argument (deprecated in Python 3.x).
**Fix:** Заменить на `re.split(r'\s*=\s*', line, maxsplit=1)` в следующей сессии.

### SA-BUG-03 — Volumes filter `/mnt /volume /data` — не найдены тома на нестандартных путях
**Статус:** ℹ️ Design limitation
**Описание:** API overview фильтрует df output по `/mnt`, `/volume`, `/data`. Тома на других путях не показываются.
**Action:** Приемлемо для rusNAS (стандарт: /mnt/*).

### SA-BUG-04 — Scan progress bar не анимирован реалистично
**Статус:** ⚠️ Minor UX
**Описание:** Progress bar растёт случайно до 90%, потом мгновенно исчезает после `scan-now` завершения.
**Action:** Нормально для MVP. Улучшить в Фазе 2 с real-time status polling.

### SA-BUG-05 — CSS class `.sa-scan-progress-fill` vs `.sa-scan-progress-bar`
**Статус:** ✅ Исправлено (в CSS добавлен `.sa-scan-progress-bar`)
**Описание:** В HTML/JS использован класс `.sa-scan-progress-bar`, в CSS был определён `.sa-scan-progress-fill`.
**Fix:** Добавлен `.sa-scan-progress-bar` в дополнительный блок CSS.

### SA-BUG-06 — Modal window использует Cockpit Bootstrap `.modal.fade` без jQuery
**Статус:** ✅ Решено через CSS
**Описание:** Cockpit не включает jQuery/Bootstrap JS. Модал управляется через CSS классов `.in` и `display:flex`.
**Fix:** `#settingsModal { display:none; } #settingsModal.in { display:flex; }` — работает без jQuery.

### SA-BUG-07 — squarify может выдавать NaN/Infinity при нулевых значениях
**Статус:** ✅ Защита есть

**Ожидаемо** — системный дизайн. Линейная регрессия требует ≥2 точек.

**Реализовано** — кнопка RAID на дашборде ведёт на /rusnas/disks (неизменна).
