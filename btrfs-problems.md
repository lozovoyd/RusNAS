## 1. Btrfs + mdadm RAID5/6: Write Hole и Partial Parity Log

**Проблема:**
Write Hole — это состояние после аварийного отключения питания в момент неатомарной записи на RAID5/6. Если питание пропало после записи данных, но до записи обновлённого блока чётности, массив окажется в несогласованном состоянии: данные одного поколения, чётность — другого. При следующем чтении RAID не может определить, какой диск содержит корректные данные. Btrfs при этом выступает только как файловая система — она проверяет свои собственные checksums, но не может исправить повреждение на уровне RAID-чётности.

Важно: это проблема **mdadm RAID5/6**, а **не Btrfs**. Btrfs с профилем `single` поверх mdadm-массива увидит повреждение через scrub (не-совпадение checksum) и сообщит об ошибке, но не сможет восстановить данные, так как под ним нет избыточности (Btrfs видит mdadm-устройство как единый диск).

**Риск:** Критический (только для RAID5/6; RAID1/10 не затронуты)

**Решение:**
Три варианта (от лучшего к допустимому):

1. **Включить PPL (Partial Parity Log)** — встроенный механизм mdadm для RAID5, хранит частичную чётность в метаданных каждого диска массива. Не требует отдельного диска-журнала:
   ```bash
   # При создании нового массива:
   mdadm --create /dev/md0 --level=5 --raid-devices=3 \
     --consistency-policy=ppl /dev/sd[bcd]

   # Для существующего массива (сначала остановить):
   mdadm --stop /dev/md0
   mdadm --assemble /dev/md0 --consistency-policy=ppl /dev/sd[bcd]
   ```
   Ограничения PPL: снижает скорость записи на 30–40%, не защищает от потери данных в полёте (только от тихого повреждения). Поддерживает только md metadata v1.x.

2. **Включить RAID5 journal cache** — отдельный SSD-диск как журнал:
   ```bash
   mdadm --grow /dev/md0 --add-journal /dev/sde
   ```
   Это полноценная защита от write hole. Дополнительный SSD — точка отказа, но не катастрофического: при его потере массив переходит в режим "no-journal" и нужен resync.

3. **UPS + корректное завершение** — простейшая защита. NUT-демон на системе уже настроен; при срабатывании UPS shutdown гарантирует чистое размонтирование до полного сброса чётности.

**Как попасть:** Любая запись данных в момент аварийного отключения питания (выдернули кабель, ИБП не успел). При любом открытом файле на RAID5/6.

## 2. CoW + mdadm RAID: деградация производительности случайной записи

**Проблема:**
Copy-on-Write в Btrfs означает, что каждая запись (даже обновление 1 байта) создаёт новый экстент в другом месте диска, а не перезаписывает существующий. Это приводит к:

- **Write amplification**: небольшая запись генерирует полный 128 KB I/O-блок Btrfs
- **Фрагментация**: со временем файлы рассеиваются по диску, последовательные чтения деградируют в случайные
- **Двойная нагрузка на RAID5/6**: CoW-паттерн Btrfs создаёт множество небольших случайных записей → mdadm RAID5/6 вынужден выполнять Read-Modify-Write для каждого stripe → производительность падает нелинейно
- **Усугубление со снапшотами**: чем больше снапшотов, тем сильнее фрагментация

На HDD-массивах это наиболее критично. Случайный доступ на HDD — самая медленная операция; CoW + RAID5/6 в сочетании могут привести к падению записи до 5–15 MB/s там, где ожидается 100+ MB/s.

**Риск:** Высокий (особенно RAID5/6 на HDD с активной записью)

**Решение:**
- Для VM-образов, баз данных и других файлов с интенсивной случайной перезаписью: **отключать CoW** атрибутом на файл или директорию:
  ```bash
  # Для новой директории — атрибут C (no-COW):
  chattr +C /mnt/data/vm-images/
  # Для файлов нужно пересоздать файл после установки атрибута
  ```
  Caveat: `nodatacow` отключает также checksumming для этих файлов.

- Использовать `autodefrag` mount-опцию с осторожностью (см. п. 14)
- Мониторить фрагментацию через `btrfs filesystem defragment -n /mnt/data` (dry-run, не дефрагментирует)
- Для NAS-нагрузки (большие файлы, документы) — CoW выгоден; для мелких случайных I/O — отключать

**Как попасть:** Любая NAS-шара, активно используемая для VM-дисков, баз данных Samba CTDB, Time Machine бэкапов (macOS).

## 3. Btrfs scrub vs. mdadm check: конфликт при параллельном запуске

**Проблема:**
Btrfs scrub и mdadm check — разные операции, читающие весь массив последовательно:

- `btrfs scrub`: читает все экстенты, верифицирует checksum'ы данных и метаданных, исправляет обнаруженные повреждения (если есть избыточность)
- `mdadm --check` (aka `check` action через sysfs): верифицирует чётность RAID5/6 (или зеркала RAID1/10), исправляет расхождения

Запуск обеих операций **одновременно** создаёт огромную I/O-нагрузку — оба процесса читают весь массив сериализованно. На реальных NAS-системах это приводило к полной неотзывчивости системы на несколько часов (задокументировано в форуме OpenMediaVault: "BTRFS-Scrub and MDADM check running in parallel -> NAS unresponsive").

Дополнительная тонкость: если mdadm check запущен в момент, когда Btrfs активно записывает, check может зафиксировать ложные "mismatches" из-за гонки между записью данных и чётности.

**Риск:** Средний (производительность) + Низкий (ложные предупреждения)

**Решение:**
Разнести расписание. Рекомендуемая схема:

```
# /etc/cron.d/rusnas-maintenance

# mdadm check — каждую первую пятницу месяца, 02:00
0 2 1-7 * * 5   root  echo check > /sys/block/md0/md/sync_action

# Btrfs scrub — каждое 15-е число месяца, 02:00 (гарантированно НЕ совпадает с mdadm)
0 2 15 * *      root  btrfs scrub start -Bd /mnt/data
```

Для ограничения скорости mdadm check (чтобы не убивал производительность):
```bash
# Максимум 50 MB/s для sync операций mdadm
echo 50000 > /proc/sys/dev/raid/speed_limit_max
echo 10000 > /proc/sys/dev/raid/speed_limit_min
```

Для Btrfs scrub с низким приоритетом (опция `-c 3 -n 4` — ionice idle):
```bash
btrfs scrub start -Bd -c 3 -n 4 /mnt/data
```

**Как попасть:** Если оба задания поставить в cron на одно и то же время/день — что легко сделать случайно.

## 4. Btrfs balance: когда запускать и как безопасно

**Проблема:**
`btrfs balance` — тяжёлая операция, перераспределяющая block groups. Она необходима в ряде ситуаций, но выполняется неверно чаще всего в двух сценариях:

1. **ENOSPC при наличии свободного места**: Btrfs может отказывать в записи с ошибкой "No space left" при `df` показывающем 30% свободно. Причина — внутренняя фрагментация: все `unallocated` слоты заняты блок-группами DATA, но нужны слоты для METADATA. `btrfs filesystem usage /mnt/data` покажет это: `Unallocated: 0`.
2. **Balance во время RAID-resync**: запуск balance пока mdadm перестраивает массив — двойная I/O-нагрузка, риск таймаутов.

Полный баланс (`btrfs balance start /mnt/data` без фильтров) **перезаписывает ВСЕ данные** — это может занять 12–48 часов на большом массиве и создаёт риск потери данных при сбое в процессе.

**Риск:** Высокий (если запускать без фильтров или во время resync)

**Решение:**

Запускать только с фильтрами и только когда необходимо:

```bash
# Диагностика — сначала смотреть статус:
btrfs filesystem usage /mnt/data

# Вариант 1: освободить фрагментированные block groups (>85% заполнены)
# Безопасный, быстрый баланс — только недозаполненные группы
btrfs balance start -dusage=85 -musage=85 /mnt/data

# Вариант 2: сжать пустые/малозаполненные DATA block groups
btrfs balance start -dusage=50 /mnt/data

# Проверить прогресс в фоне:
btrfs balance status /mnt/data

# Остановить при необходимости (можно продолжить позже):
btrfs balance cancel /mnt/data
```

**НИКОГДА не запускать** во время mdadm resync/reshape. Проверять:
```bash
cat /proc/mdstat  # убедиться что нет "resync" или "reshape" в выводе
```

**Плановый цикл для NAS:**
- Запускать balance раз в месяц с `dusage=85` — превентивная мера
- Запускать balance сразу после добавления дисков в массив
- **Не запускать** когда свободного места меньше 20 GB

**Как попасть:** Запустить полный `btrfs balance start /mnt/data` без фильтров через Cockpit UI — операция перепишет все данные и может занять сутки.

## 5. Отключение питания и восстановление после краша

**Проблема:**
При аварийном отключении питания (без UPS или при слишком быстром разряде) происходит несколько событий последовательно:

1. **mdadm RAID5/6**: незаписанная чётность → write hole (см. п. 1)
2. **Btrfs journal replay**: после монтирования Btrfs читает log-tree и воспроизводит незафиксированные транзакции. Если log-tree повреждён — монтирование провалится с ошибкой `kernel: BTRFS error: block xxx is not a node`
3. **Метаданные Btrfs**: суперблок Btrfs хранит несколько поколений (2 копии), шансы на восстановление высокие, но не 100%

Известный баг в kernel 6.15.3+ (не в 6.12 LTS): "log tree replay failure due to file with 0 links and extents" — повреждение при воспроизведении журнала после краша. В kernel 6.12 LTS этот баг отсутствует (исправлен в другой ветке).

Специфика для mdadm: перед тем как Btrfs сможет replay journal, mdadm должен завершить resync (для RAID5/6 — полный проход массива). Это может занять **несколько часов** — всё это время данные недоступны.

**Риск:** Критический (потеря данных при стечении обстоятельств) / Высокий (длительная недоступность)

**Решение:**

- **UPS + NUT**: уже настроен в системе. Убедиться что `upsmon.conf` содержит:
  ```
  POWERDOWNFLAG /etc/killpower
  SHUTDOWNCMD "/sbin/shutdown -h +0"
  ```
  и интервал между "battery low" и командой shutdown достаточен (минимум 60 сек).

- При повреждённом Btrfs после краша — процедура восстановления:
  ```bash
  # 1. Убедиться что mdadm-массив в чистом состоянии
  mdadm --detail /dev/md0 | grep State
  # Должно быть: clean

  # 2. Попробовать монтирование с recovery-опциями
  mount -o ro,rescue=usebackuproot /dev/md0 /mnt/recovery

  # 3. Если не помогло — использовать backup superblock
  btrfs rescue super-recover /dev/md0

  # 4. Проверить и починить log tree
  btrfs rescue zero-log /dev/md0
  # ВНИМАНИЕ: zero-log теряет незакоммиченные транзакции — это допустимо после краша
  ```

- Для ускорения mdadm resync после перезагрузки:
  ```bash
  echo 200000 > /proc/sys/dev/raid/speed_limit_max
  ```

**Как попасть:** Внезапное отключение питания без UPS. Принудительный reboot сервера без остановки.

## 6. Btrfs-снапшоты + mdadm RAID: фрагментация и ограничения

**Проблема:**
Btrfs-снапшоты работают корректно поверх mdadm RAID любого уровня — нет прямых конфликтов на уровне API. Однако есть операционные риски:

1. **Фрагментация нарастает со временем**: каждый снапшот "замораживает" экстенты. При последующих записях Btrfs вынужден писать данные в новые места, не затрагивая общие экстенты снапшота. После 12+ снапшотов на один сабволюм производительность HDD падает драматически — все read-операции превращаются в случайные.

2. **Дефрагментация снапшотов опасна**: `btrfs defrag` на родительском субтоме разрушает shared extents со снапшотами — их размер на диске увеличивается до размера полной копии каждого снапшота. На массиве с 50 снапшотами дефрагментация может мгновенно заполнить диск.

3. **Восстановление снапшота**: при восстановлении файла из снапшота (reflink-копия) на mdadm RAID5/6 — операция безопасна, но при аварийном отключении в процессе — write hole риск остаётся.

4. **Место на диске**: снапшоты не занимают место сразу, но при удалении оригинала место не освобождается пока есть снапшоты. На RAID5/6 при переполнении mdadm-массива и одновременном накоплении снапшотов — ENOSPC без предупреждения.

**Риск:** Средний (производительность) / Высокий (дисковое пространство)

**Решение:**
- **Ограничить количество снапшотов**: не более 12–15 на один субтом. Текущая система `rusnas-snap` с расписанием hourly/daily/weekly — убедиться что retention policy удаляет старые снапшоты.
- **Никогда не запускать `btrfs defrag` на субтоме у которого есть снапшоты** (см. п. 14)
- **Мониторить реальный размер**:
  ```bash
  btrfs filesystem du -s /mnt/data/.snapshots/
  ```
- **Хранить минимум 15% свободного места** — с запасом на снапшотный overhead

**Как попасть:** Настроить hourly снапшоты без автоматического удаления → через месяц 700+ снапшотов → HDD-массив зависает при любом доступе.

## 7. Transparent HugePages + Btrfs

**Проблема:**
Transparent HugePages (THP) по умолчанию включены (`enabled=always`) в большинстве дистрибутивов. Для NAS-нагрузки это создаёт несколько проблем:

1. **Khugepaged и Btrfs page cache**: THP может пытаться объединять страницы page cache Btrfs в 2MB hugepages. При интенсивной записи это приводит к задержкам (khugepaged останавливает процесс для "collapse" страниц).

2. **Memory pressure + defrag**: при нехватке памяти THP-defrag пытается перемещать страницы для создания контiguоus 2MB blocks — это конкурирует с I/O Btrfs за Page Cache Lock, увеличивая latency.

3. **Документированные проблемы с хранилищами**: Red Hat, Veritas, TiDB и другие производители хранилищ рекомендуют отключать THP для storage workloads. Конкретно для Btrfs: увеличенные latency spike'и при интенсивном SMB/NFS трафике.

**Риск:** Средний (нерегулярные latency spike'и, деградация при memory pressure)

**Решение:**
Переключить в режим `madvise` вместо полного отключения — безопаснее для остальных компонентов:

```bash
# Немедленно (до перезагрузки):
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled
echo never   > /sys/kernel/mm/transparent_hugepage/defrag

# Постоянно (через systemd-tmpfiles или rc.local):
# /etc/tmpfiles.d/thp-nas.conf
w /sys/kernel/mm/transparent_hugepage/enabled - - - - madvise
w /sys/kernel/mm/transparent_hugepage/defrag  - - - - never
```

Это уже учтено в `rusnas-performance-tuning-spec.md` (`echo madvise > ...`). Убедиться что deploy Performance Tuner применяет эти настройки.

**Как попасть:** Дефолтная конфигурация Debian 13. Проявляется при >4 активных SMB-клиентах с интенсивным трафиком.

## 8. mdadm stripe_cache_size: настройка для RAID5/6 + Btrfs

**Проблема:**
`stripe_cache_size` — количество страниц памяти (по 4 KB каждая), выделяемых mdadm для кеширования stripe-операций RAID5/6. Default: **256 страниц = 1 MB**.

Для Btrfs это критично: Btrfs пишет блоками по 128 KB, mdadm stripe обычно = `chunk_size * (N-1)` дисков. При несоответствии размеров: mdadm вынужден выполнять **Read-Modify-Write** для каждого неполного stripe. При маленьком stripe_cache это происходит с чтением с диска (не из кеша) → производительность записи на RAID5/6 падает в 3–5 раз.

Формула оценки: `stripe_cache_size = (активные_записи * stripe_size_pages)`. Для 3-дискового RAID5 с chunk=512K, stripe = 512K * 2 = 1MB = 256 страниц. При 16 одновременных писателях нужно минимум 4096 страниц.

**Риск:** Высокий (производительность записи RAID5/6)

**Решение:**

```bash
# Смотреть текущее значение:
cat /sys/block/md0/md/stripe_cache_size

# Установить оптимальное значение (4096 = 16MB — рекомендация для NAS)
echo 4096 > /sys/block/md0/md/stripe_cache_size

# Для очень нагруженных систем (10GbE, много клиентов):
echo 8192 > /sys/block/md0/md/stripe_cache_size
# 8192 страниц × 4KB = 32 MB RAM — приемлемо для NAS с 8+ GB RAM

# Сделать постоянным через /etc/rc.local или systemd unit:
# /etc/systemd/system/mdadm-stripe-cache.service
# ExecStart=/bin/sh -c 'for md in /sys/block/md*/md/stripe_cache_size; do echo 4096 > $md; done'
```

**Зависимость от RAM**: каждые 1024 страницы = 4 MB RAM × количество md-устройств. На системе с 8 GB RAM значение 4096 безопасно.

**Как попасть:** Дефолтное значение 256 при создании массива через `mdadm --create`. Никогда не упоминается в стандартных гайдах.

## 9. noatime/relatime: fstab-параметры для Btrfs на NAS

**Проблема:**
Btrfs по умолчанию использует `relatime` — обновляет access time при первом чтении файла (если atime старше 24ч или новее mtime). Для CoW-filesystem это **катастрофически дорого**:

- Каждое обновление atime = **новый CoW-коммит метаданных**
- При 100 одновременных SMB-клиентах, читающих файлы → 100 × N CoW-записей в метаданные только из-за atime
- "Худший случай relatime" — после создания снапшота все файлы имеют atime старше 24ч. Первое чтение каждого файла генерирует atime-обновление → массовый CoW-шторм метаданных

Официальная документация Btrfs явно указывает: `noatime` сильно рекомендуется для любого production-использования.

**Риск:** Средний (производительность, I/O amplification)

**Решение:**

`/etc/fstab` для Btrfs-массива на NAS:

```fstab
UUID=<uuid>  /mnt/data  btrfs  defaults,noatime,space_cache=v2,compress=zstd:3  0  0
```

Полный рекомендуемый набор mount-опций:

| Опция | Причина |
|-------|---------|
| `noatime` | Устраняет CoW-запись при каждом чтении |
| `space_cache=v2` | Более эффективный кеш свободного пространства (default в kernel 5.15+) |
| `compress=zstd:3` | Компрессия данных (см. п. 10) |
| `autodefrag` | Автодефрагментация мелких файлов (только для HDD, с оговорками) |
| `discard=async` | Async TRIM для SSD (не применять к HDD mdadm-массивам) |

**ВАЖНО**: `autodefrag` **не использовать** если есть снапшоты с общими экстентами — разрушает sharing (см. п. 14).

**Как попасть:** Дефолтный `mount` без явных опций, или `/etc/fstab` с `defaults`.

## 10. Сжатие zstd + mdadm RAID: взаимодействие и производительность

**Проблема:**
Btrfs сжимает данные блоками по **128 KB** перед записью. Это создаёт специфичное взаимодействие с mdadm:

1. **Неполный stripe write**: если сжатый блок меньше stripe-размера mdadm, возникает Read-Modify-Write на RAID5/6. Это нормально, но увеличивает write latency.

2. **Уже сжатые файлы**: медиа-контент (JPEG, MP4, MP3, архивы ZIP/7z) не сжимается повторно — Btrfs определяет это после нескольких неудачных попыток и пишет данные без сжатия. CPU overhead минимален для таких файлов.

3. **compress vs compress-force**: `compress-force` пытается сжать **все** файлы включая уже сжатые — приводит к write amplification (сжатый файл распухает) и лишнему CPU. **Не использовать** `compress-force` на NAS с медиа-контентом.

4. **Уровни zstd**: более высокий уровень = лучше сжатие, но дольше сжатие/распаковка. На NAS-нагрузке (сотни клиентов) CPU становится узким местом при zstd:9+.

**Риск:** Низкий (правильно настроенное сжатие безопасно)

**Решение:**

Рекомендация для RusNAS (HDD-массив):
```
compress=zstd:3
```

| Тип контента | Рекомендация |
|--------------|--------------|
| Документы, текст, БД | `zstd:3` — хорошее сжатие, приемлемая скорость |
| Медиа (фото/видео) | Сжатие не помогает; Btrfs сам определит и пропустит |
| VM-образы с `nodatacow` | Сжатие не применяется вместе с `nodatacow` |
| Бэкап-архивы | `zstd:1` — минимальная нагрузка, уже сжатые пропускаются |

Не использовать `compress-force` — только `compress=zstd:3`.

Применить сжатие к существующим данным retroactively:
```bash
btrfs filesystem defragment -r -czstd /mnt/data/shares/
# НЕ применять к субтомам со снапшотами — разрушает shared extents!
```

**Как попасть:** Использование `compress-force=zstd` вместо `compress=zstd` в fstab.

## 11. Btrfs quota groups (qgroups): известные баги

**Проблема:**
Btrfs qgroups — механизм учёта и ограничения дискового пространства по субтомам. Официальная документация предупреждает: "активация qgroups не рекомендуется если не планируется их использовать" — из-за серьёзных накладных расходов.

Известные проблемы:

1. **Системный вызов `sys_path_to_bdev()` падает с Btrfs + qgroups**: macOS-клиенты при попытке записи больших файлов через SMB получают "unexpected error" — именно из-за того что Btrfs не поддерживает POSIX-квоты через Linux quota API, а macOS требует информацию о квотах для проверки доступного места. **Решение из bugs.md**: использовать `vfs objects = catia fruit streams_xattr` в Samba + fruit-модуль.

2. **Некорректный учёт** при сложных сценариях снапшотов (nested subvolumes, reflinks между субтомами). Значение "exclusive" bytes может быть неверным.

3. **Производительность**: при включённых qgroups **каждая операция с экстентами** выполняет backreference walk — O(N) по числу снапшотов. При 100+ снапшотах это вызывает заметные паузы.

4. **Simple Quotas (squotas)**: kernel 6.7+ добавил упрощённый режим квот без backreference walk — только "первый записавший владеет". Не учитывает shared extents, но производительность в 10× лучше.

**Риск:** Высокий (если включены без необходимости) / Низкий (если отключены)

**Решение:**

```bash
# Проверить статус:
btrfs quota show /mnt/data

# Отключить если не нужны:
btrfs quota disable /mnt/data

# Если нужен учёт по субтомам — использовать simple quotas (kernel 6.7+):
btrfs quota enable --simple /mnt/data
```

**Не включать qgroups** только для получения размеров снапшотов — для этого использовать:
```bash
btrfs filesystem du -s /mnt/data/.snapshots/<snap-name>
```

**Как попасть:** Cockpit Storage Analyzer включает qgroups для вычисления размеров субтомов. Убедиться что код использует `btrfs subvolume show` или `du`, а не `btrfs qgroup show`.

## 12. Отказ диска + mdadm rebuild: поведение Btrfs

**Проблема:**
При отказе диска в mdadm RAID5/6 и последующем rebuild:

1. **Период rebuild**: массив работает в degraded-режиме. Производительность записи падает на 30–60% (mdadm вычисляет чётность для восстановления). Btrfs продолжает работать нормально — он не знает о деградации RAID.

2. **Btrfs scrub во время rebuild**: запуск scrub во время mdadm rebuild создаёт конкурирующую полную нагрузку на I/O. Btrfs может обнаружить "ошибки checksum" на данных, которые mdadm ещё не восстановил (не записал на новый диск). Scrub зафиксирует их как uncorrectable errors. **Не запускать scrub во время rebuild**.

3. **Дважды degraded RAID1**: известный Debian-баг 2025: при двукратном монтировании RAID1 в деградированном режиме Btrfs переходит в read-only. Для RusNAS (RAID5/6 поверх mdadm) этот конкретный баг не применим — mdadm обеспечивает уровень избыточности, Btrfs видит только одно устройство.

4. **Длительность rebuild**: mdadm rebuild на RAID5 3×4TB @ 100 MB/s = ~11 часов. В это время данные не резервированы.

**Риск:** Высокий (window of vulnerability), Средний (ложные Btrfs ошибки)

**Решение:**

Процедура при отказе диска:
```bash
# 1. Зафиксировать отказавший диск (не размонтировать Btrfs)
mdadm --detail /dev/md0  # смотреть State

# 2. Если диск ещё в массиве — пометить как failed
mdadm --fail /dev/md0 /dev/sdb

# 3. Удалить из массива
mdadm --remove /dev/md0 /dev/sdb

# 4. Добавить новый диск (начнётся rebuild автоматически)
mdadm --add /dev/md0 /dev/sdb

# 5. ЗАПРЕТИТЬ scrub до завершения rebuild:
# Мониторить прогресс:
watch -n 5 cat /proc/mdstat

# 6. По завершении rebuild — запустить scrub:
btrfs scrub start -Bd /mnt/data
```

Ускорить rebuild (если нет клиентов):
```bash
echo 200000 > /proc/sys/dev/raid/speed_limit_max
```

**Как попасть:** Запустить плановый `btrfs scrub` по расписанию который попадает на период rebuild.

## 13. mdadm write-intent bitmap + Btrfs

**Проблема:**
mdadm write-intent bitmap (internal bitmap) ускоряет resync после сбоя питания — запоминает "грязные" области и при следующем старте перепроверяет только их, а не весь массив. Это критически важно для быстрого восстановления после сбоя.

Однако у bitmap есть конфликт с Btrfs CoW-паттерном:

- Btrfs никогда не перезаписывает данные на месте → каждая запись всегда в новое место диска
- Internal bitmap отслеживает "было ли записано" на каждые 512KB/1MB/2MB региона
- CoW означает, что **весь диск постоянно "помечается как dirty"** bitmap'ом
- При RAID5/6 + много клиентов: bitmap chunk становится постоянно "active" → накладные расходы на сброс bitmap в метаданные растут
- Задокументировано: с включённым bitmap случайный write IOPS на RAID-массиве падает на 30–50% при мелких блоках

**Риск:** Средний (производительность случайной записи)

**Решение:**

Для NAS RusNAS bitmap **рекомендуется оставить включённым** — преимущество (быстрый resync) перевешивает потерю производительности при типичной NAS-нагрузке (крупные файлы, последовательная запись). Отключать только если:
- Система нагружена VMs или базами данных (много мелких случайных записей)
- Нет ИБП (bitmap всё равно не поможет при внезапном сбое питания)

```bash
# Проверить наличие bitmap:
mdadm --detail /dev/md0 | grep Bitmap

# Отключить (только если подтверждена проблема с производительностью):
mdadm --grow /dev/md0 --bitmap=none

# Увеличить chunk size bitmap для снижения overhead (256MB вместо default):
mdadm --grow /dev/md0 --bitmap-chunk=262144
```

**Как попасть:** Создать RAID-массив с default bitmap, запустить нагрузку VMs через Btrfs-шару — производительность IOPS падает вдвое.

## 14. Btrfs defrag: безопасно ли на mdadm RAID?

**Проблема:**
`btrfs filesystem defragment` — самая опасная "полезная" операция в арсенале Btrfs. Она безопасна относительно самого mdadm, но **разрушительна для shared extents**:

- Дефрагментация файла, у которого есть снапшоты или reflink-копии → файл перезаписывается в новое место → связь со снапшотом/reflink разрушается → данные дублируются → дисковое пространство может умножиться в разы
- На NAS с регулярными снапшотами (как в RusNAS) — это означает: дефрагментация ANY файла может привести к тому, что место на диске будет занято полными копиями снапшотов

Пример: субтом с 1000 файлами и 10 снапшотами. Дефрагментация → у каждого из 10 снапшотов теперь полная независимая копия → 10× увеличение потребления места.

**Риск:** Критический (при использовании снапшотов), Низкий (без снапшотов)

**Решение:**

**ПРАВИЛО**: никогда не запускать `btrfs defrag` на субтоме если есть снапшоты.

Безопасные сценарии:
```bash
# БЕЗОПАСНО: дефрагментация ТОЛЬКО если снапшотов нет
# Проверить наличие снапшотов:
btrfs subvolume list /mnt/data | grep snap

# БЕЗОПАСНО: дефрагментация конкретного файла, не имеющего CoW-копий:
btrfs filesystem defragment -v /mnt/data/shares/public/largefile.iso

# БЕЗОПАСНО: конвертировать compression на файле (с --compress):
btrfs filesystem defragment -czstd /mnt/data/shares/documents/report.docx

# НЕБЕЗОПАСНО: рекурсивная дефрагментация директории со снапшотами:
# btrfs filesystem defragment -r /mnt/data/  ← НЕ ДЕЛАТЬ
```

**autodefrag** mount-опция: включает автоматическую дефрагментацию мелких файлов (<64KB последовательных блоков). Безопасна для workload без снапшотов, **небезопасна при активном использовании снапшотов**.

**Как попасть:** Запустить `btrfs filesystem defragment -r /mnt/data` "для оптимизации" после жалоб на медленное чтение — при наличии месячных снапшотов заполнить диск за минуты.

## 15. Расширение RAID-массива + btrfs resize: процедура и ловушки

**Проблема:**
Расширение RAID-массива (добавление диска или замена дисков на большие) — одна из самых длительных и уязвимых операций. Конкретные риски:

1. **Reshape vulnerability window**: во время reshape (перераспределения данных по новому количеству дисков) массив **уязвим к потере данных** при следующем отказе диска. На 3×4TB это может занять 10–20 часов.

2. **Bitmap перед grow**: если включён write-intent bitmap, его необходимо **удалить перед grow и восстановить после** — иначе grow завершится ошибкой или bitmap станет некорректным.

3. **Btrfs не увидит новое место автоматически**: после завершения mdadm reshape файловая система Btrfs знает только об оригинальном размере устройства. Необходимо явно вызвать `btrfs filesystem resize`.

4. **Неполная последовательность**: если сделать `mdadm --grow` но не сделать `btrfs filesystem resize` — место есть на уровне блочного устройства, но Btrfs его не использует. `df` покажет старый размер.

5. **Прерванный reshape**: reshape можно прервать и продолжить позже (он сохраняет прогресс в метаданных массива). Прерванный reshape при перезагрузке продолжается автоматически — это нормальное поведение.

**Риск:** Высокий (уязвимость во время reshape)

**Решение:**

Полная безопасная процедура расширения RAID5 (добавление 4-го диска):

```bash
# === ШАГ 1: ПОДГОТОВКА ===
# Убедиться что массив чистый:
mdadm --detail /dev/md0 | grep State
# Должно быть: clean

# Создать снапшот критических данных (если возможно):
btrfs subvolume snapshot /mnt/data /mnt/data/.snapshots/pre-expand-$(date +%Y%m%d)

# === ШАГ 2: УДАЛИТЬ BITMAP (обязательно перед grow) ===
mdadm --grow /dev/md0 --bitmap=none

# === ШАГ 3: ДОБАВИТЬ ДИСК ===
mdadm --add /dev/md0 /dev/sde

# === ШАГ 4: ЗАПУСТИТЬ RESHAPE ===
# Это ДОЛГАЯ операция (часы) — массив остаётся mounted и доступным
mdadm --grow /dev/md0 --raid-devices=4

# Мониторить прогресс:
watch -n 10 cat /proc/mdstat

# === ШАГ 5: ДОЖДАТЬСЯ ЗАВЕРШЕНИЯ ===
# /proc/mdstat должен показать "active raid5" без "reshape"

# === ШАГ 6: ВОССТАНОВИТЬ BITMAP ===
mdadm --grow /dev/md0 --bitmap=internal

# === ШАГ 7: РАСШИРИТЬ BTRFS ===
# Вариант A: расширить до максимума:
btrfs filesystem resize max /mnt/data

# Вариант B: до конкретного размера:
btrfs filesystem resize 12T /mnt/data

# Проверить результат:
btrfs filesystem usage /mnt/data
df -h /mnt/data
```

**Важно для Cockpit UI (disks.js)**: функция `checkReshapeComplete()` мониторит `/proc/mdstat` и вызывает `btrfs filesystem resize max` автоматически — убедиться что эта логика корректно определяет завершение reshape (проверять отсутствие `reshape` И `resync` keyword, не только одного из них — известный баг из CLAUDE.md).

**Как попасть:** Запустить `mdadm --grow` с bitmap, получить ошибку. Или завершить grow но не сделать btrfs resize — место "пропало".

## Минимальный чеклист для нового развёртывания RusNAS

```bash
# 1. fstab: noatime + space_cache=v2
UUID=xxx /mnt/data btrfs defaults,noatime,space_cache=v2,compress=zstd:3 0 0

# 2. stripe_cache_size для всех md-устройств
for md in /sys/block/md*/md/stripe_cache_size; do echo 4096 > $md; done

# 3. THP в madvise режим
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled
echo never   > /sys/kernel/mm/transparent_hugepage/defrag

# 4. Ограничение скорости sync (не убивать производительность во время rebuild)
echo 50000 > /proc/sys/dev/raid/speed_limit_max

# 5. PPL для RAID5 (при создании нового массива)
# mdadm --create /dev/md0 --level=5 --consistency-policy=ppl ...

# 6. Расписание обслуживания (разнести scrub и mdadm check)
# Scrub: 15-е число каждого месяца
# mdadm check: 1-я пятница каждого месяца

# 7. qgroups отключены
btrfs quota disable /mnt/data

# 8. Retention policy снапшотов: не более 12-15 на субтом
```
