# rusNAS Distribution, Licensing & Source Protection — Design Spec
**Date:** 2026-03-26
**Status:** Approved
**Scope:** Update repository, device installation, source code protection, license enforcement

---

## 1. Overview

Three interconnected subsystems built on a single VPS (Подход 1 — монолитный):

| Subsystem | What it solves |
|-----------|---------------|
| **apt repository** | Distributes updates to 1000+ devices; gated by license |
| **Bootstrap installer** | One-command install on clean Debian 13 |
| **Source protection** | JS minification + Python .pyc; closed git repo |

All three use the existing license server architecture (`rusnas-license-server-task.md`) as their foundation.

## 2. Infrastructure: Single VPS

```
VPS: activate.rusnas.ru
├── nginx
│   ├── /                    → FastAPI License Server (:8765)
│   ├── /admin/*             → License Admin Web UI (Jinja2, protected)
│   ├── /apt/                → reprepro apt repository
│   │   ├── trixie/main/     → feature updates (requires updates_features)
│   │   └── trixie/security/ → security patches (any active license)
│   ├── /install             → bootstrap.sh
│   └── /apt-auth-internal   → internal auth_request endpoint (:8766)
├── rusnas_licenses.db        (SQLite, shared between license server + apt-auth)
└── operator_private.pem
```

**Rationale:** SQLite + reprepro handles 1000+ devices comfortably. Single machine simplifies ops. Scale to separate services only if apt repo exceeds tens of GB/day.

## 4. Serial Number Flow

Serials are **pre-generated by the operator** and loaded into SQLite before shipping devices. The device never generates its own serial.

```
Operator:
  python3 admin.py serial import batch-2026-03.txt
    → 100 serials added to SQLite: issued_at = now(), first_install_at = NULL

Physical device / order:
  → serial printed on sticker or included in order email

Customer installs:
  curl -fsSL https://activate.rusnas.ru/install | bash -s -- RUSNAS-XXXX-XXXX-XXXX-XXXX
    → bootstrap writes serial to /etc/rusnas/serial
    → bootstrap calls GET /api/install-token?serial=RUSNAS-...
    → server sets first_install_at = now() (if NULL), returns signed Ed25519 install-token
    → token saved to /etc/apt/auth.conf.d/rusnas.conf as the apt password

Customer activates license:
    → opens activate.rusnas.ru, enters serial, gets RNAC-code
    → pastes code into Cockpit → Лицензия
    → rusnas-license daemon verifies + replaces apt password with raw activation code
```

## 5. Grace Period & Update Gating

**First 6 months after install: all updates free, regardless of license.**
**After 6 months: license gates access.**

### Anti-tamper: server-side clock only

The `install_at` timestamp is recorded server-side in `serials.first_install_at` at first bootstrap call. The apt-auth validator uses the **server's own `time.time()`** — never the device clock. Changing the device date has zero effect.

### Token types

There are two valid token formats for the apt `password` field. Both are Ed25519-signed blobs produced by `sign_payload()` and verified by `verify_and_decode()`. The `type` field distinguishes them:

| Field | `install` token | `activation` token |
|-------|----------------|--------------------|
| `ver` | `1` | `1` |
| `type` | `"install"` | `"activation"` (existing spec payload) |
| `serial` | ✓ | ✓ |
| `install_at` | ✓ (from `serials.first_install_at`) | — |
| `issued_at` | — | ✓ (moment code was signed) |
| `license_type` | — | ✓ |
| `expires_at` | — | ✓ |
| `features` | — | ✓ |
| `customer` | — | ✓ |
| `max_volumes` | — | ✓ |

The activation token format is defined in `rusnas-license-server-task.md` §5. The install token is a new, simpler format defined here.

### apt password lifecycle

| Phase | apt `password` value |
|-------|---------------------|
| Post-bootstrap, pre-activation | Signed install-token (Base62, ~100 chars) |
| Post-activation | Raw Base62 activation code (~300–380 chars, full RNAC payload without formatting) |

`auth.conf` netrc format supports passwords of any length — 300-char passwords are valid.

### apt-auth.py logic

```python
def check_access(authorization_header, request_uri):
    # authorization_header = "Basic base64(login:password)"
    serial, token = decode_basic_auth(authorization_header)

    payload = verify_and_decode(public_key, token)   # Ed25519 — cannot be forged
                                                      # raises InvalidSignature if tampered

    # Verify serial in token matches claimed serial
    if payload["serial"] != serial:
        return DENY

    token_type = payload.get("type", "activation")

    # Determine grace period start
    if token_type == "install":
        install_at = payload["install_at"]
    else:
        # activation token: look up first_install_at from DB (server recorded at bootstrap)
        row = db.get_serial(serial)
        install_at = row["first_install_at"]
        if install_at is None:
            return DENY  # first_install_at NULL = DB inconsistency; deny rather than grant fresh grace

    grace_end = install_at + 180 * 86400             # SERVER clock, tamper-proof
    now = time.time()                                # SERVER clock

    if now < grace_end:
        return ALLOW                                 # grace period: all components open

    # After grace period — check license from DB (activation token features not trusted alone)
    license = db.get_active_license(serial)
    component = extract_component_from_uri(request_uri)  # "main" or "security"

    if component == "security":
        return ALLOW if license and not license["revoked"] else DENY
    if component == "main":
        return ALLOW if license_has_active_feature(license, "updates_features") else DENY
    return DENY
```

**Note on revocation cache:** nginx caches `auth_request` responses for 60 seconds. Revoked licenses will continue to have apt access for up to 60 seconds after revocation. This is acceptable for this use case — apt access lockout is not an instant-response security mechanism.

### apt-auth service (`:8766`)

Separate lightweight FastAPI process. Shares the same SQLite DB (read-only access).
nginx calls it via `auth_request /apt-auth-internal`.

```nginx
location /apt/ {
    auth_request /apt-auth-internal;
    error_page 401 403 = @apt_denied;
    root /var/lib/reprepro/public;
    proxy_cache_valid 200 60s;
}

location = /apt-auth-internal {
    internal;
    proxy_pass http://127.0.0.1:8766;
    proxy_pass_request_body off;
    proxy_set_header Content-Length "";
    proxy_set_header Authorization $http_authorization;    # REQUIRED: forward apt credentials
    proxy_set_header X-Original-URI $request_uri;         # component detection
}
```

## 6. License Server Additions

Beyond the existing `rusnas-license-server-task.md`, add:

### Schema change: `serials` table

Add `first_install_at INTEGER` (nullable) to the existing `serials` table:

```sql
ALTER TABLE serials ADD COLUMN first_install_at INTEGER;
-- NULL = serial issued but device not yet bootstrapped
-- non-NULL = timestamp of first /api/install-token call (server time)
```

`issued_at` remains `NOT NULL` (set by `admin.py serial import` or `admin.py serial add`).

### New endpoint: `GET /api/install-token`

```
GET /api/install-token?serial=RUSNAS-XXXX-XXXX-XXXX-XXXX

Logic:
1. Validate serial format (same restricted alphabet as models.py: [BCDFGHJKLMNPQRSTVWXYZ2345679])
2. Check serial exists in SQLite → {"ok": false, "error": "serial_not_found"} if not
3. If serials.first_install_at IS NULL → UPDATE serials SET first_install_at = now()
4. Build install token payload:
   {"ver": 1, "type": "install", "serial": "<serial>", "install_at": <first_install_at>}
5. sign_payload(private_key, payload) → raw Base62 install-token
6. audit("install_token_issued", serial, ip)
7. Return {"ok": true, "token": "<raw_base62>", "install_at": <ts>}

Rate limit: 10/hour per serial
```

### Admin Web UI: `/admin/*`

Protected by `Authorization: Bearer <ADMIN_TOKEN>` (from .env).
Server-rendered HTML with Jinja2. rusNAS dark theme (no external CDN).

| Route | Description |
|-------|-------------|
| `GET /admin/` | Dashboard: total serials, active licenses, expiring soon |
| `GET /admin/serials` | List all serials with status; import batch (textarea) |
| `GET /admin/licenses` | List licenses: active / expired / revoked filters |
| `GET /admin/license/new` | Form: serial + type + features + customer + expiry |
| `GET /admin/license/<id>` | Detail: activation history, extend, revoke |
| `GET /admin/report` | Same as `admin.py report` but in browser |

## 7. Bootstrap Installer (`bootstrap.sh`)

Served at `https://activate.rusnas.ru/install`.

```bash
curl -fsSL https://activate.rusnas.ru/install | bash -s -- RUSNAS-A3F7-KM9P-2BXQ-HJ4N
```

### Steps

```
1. PREFLIGHT
   - Require root
   - Verify Debian 13 (lsb_release -cs == "trixie")
   - Validate serial: regex ^RUSNAS-[BCDFGHJKLMNPQRSTVWXYZ2345679]{4}(-[BCDFGHJKLMNPQRSTVWXYZ2345679]{4}){3}$
   - Check internet: curl -sf https://activate.rusnas.ru/health
   - apt-get install -y jq (if missing)

2. SERIAL
   - mkdir -p /etc/rusnas
   - echo "$SERIAL" > /etc/rusnas/serial

3. INSTALL TOKEN (skip if already activated)
   - If /etc/rusnas/license.json exists:
       echo "License already active. Skipping token update."
   - Else:
       TOKEN=$(curl -sf "https://activate.rusnas.ru/api/install-token?serial=$SERIAL" | jq -r .token)
       mkdir -p /etc/apt/auth.conf.d
       Write /etc/apt/auth.conf.d/rusnas.conf:
         machine activate.rusnas.ru/apt/
         login  $SERIAL
         password $TOKEN

4. APT REPO
   - curl .../rusnas.gpg > /etc/apt/keyrings/rusnas.gpg
   - Write /etc/apt/sources.list.d/rusnas.list (idempotent — overwrite is safe)
   - apt-get update -qq

5. INSTALL
   - apt-get install -y cockpit rusnas-system

6. DONE
   - Print device IP + instructions to open Cockpit + activate license
```

**Properties:**
- Idempotent: safe to run twice — skips token update if license.json already present, overwrites sources.list safely
- Serial visible in `ps aux` during install — acceptable for an installer

## 9. Build & Release Pipeline

### Files

```
RusNAS/
├── VERSION           # e.g. 1.0.0
├── build-deb.sh      # build .deb from source
└── release.sh        # build + sign + publish to reprepro
```

### `build-deb.sh`

Uses **globs only** — no hardcoded file lists. New files added to correct folders are automatically included.

```bash
# JS: terser all *.js → *.min.js
for f in cockpit/rusnas/js/*.js; do
    terser "$f" --compress --mangle \
        -o "pkg/usr/share/cockpit/rusnas/js/$(basename ${f%.js}.min.js)"
done
sed -i 's/\.js"/\.min.js"/g' pkg/usr/share/cockpit/rusnas/*.html

# Python: compile all scripts/*.py → *.pyc (no sources)
python3 -m compileall -b cockpit/rusnas/js/scripts/
cp cockpit/rusnas/js/scripts/*.pyc pkg/usr/share/cockpit/rusnas/js/scripts/

# Static assets (auto-include all)
cp cockpit/rusnas/*.html cockpit/rusnas/manifest.json pkg/usr/share/cockpit/rusnas/
cp cockpit/rusnas/css/*.css pkg/usr/share/cockpit/rusnas/css/

# Version
sed -i "s/^Version:.*/Version: $(cat VERSION)/" pkg/DEBIAN/control
dpkg-deb --build pkg/ rusnas-system_$(cat VERSION)_amd64.deb
```

### `release.sh`

```bash
./release.sh [main|security] <version>
# e.g.:
./release.sh security 1.0.1   # security patch — all active licenses
./release.sh main 1.1.0        # feature release — requires updates_features
```

Steps: bump VERSION → `build-deb.sh` → GPG sign → `scp` to VPS → `reprepro includedeb trixie <component>`.

### Rule: security vs main

| Publish to | When |
|-----------|------|
| `security` | CVE fix, data-loss bug, auth/permission vulnerability, Guard detection fix |
| `main` | New Cockpit page, new feature, UI improvement, performance improvement |

## 10. Cockpit License Page (`license.html` + `license.js`)

New page added to the Cockpit plugin. **Requires manual entry in `manifest.json`** (HTML glob picks up the file, but Cockpit will not show the page without a manifest entry).

### UI

```
┌─────────────────────────────────────────────┐
│  🔑 Лицензия                                │
│                                             │
│  Серийный номер: RUSNAS-A3F7-KM9P-2BXQ-HJ4N│
│  Статус: ● Активна (Standard)               │
│  Клиент: ООО Ромашка                        │
│  Действует до: 01.03.2027                   │
│  Фичи: Guard ✓  Снапшоты ✓  Обновления ✓   │
│                                             │
│  [Активировать / Переактивировать]          │
│                                             │
│  ┌─ Код активации (RNAC-...) ─────────────┐ │
│  │ <textarea>                             │ │
│  └────────────────────────────────────────┘ │
│  [Применить]                                │
└─────────────────────────────────────────────┘
```

### `license.js` flow on activation

```javascript
1. normalize_activation_code(input)
   // убрать "RNAC-", все пробелы и переносы строк → raw Base62 string

2. verify_and_decode(public_key, raw_base62)
   // Ed25519 через WebCrypto API: crypto.subtle.verify("Ed25519", ...)
   // throws if signature invalid

3. assert payload.serial === readFile("/etc/rusnas/serial").trim()
   // reject if serial mismatch

4. cockpit.file("/etc/rusnas/license.json", {superuser:"require"})
   .replace(JSON.stringify(payload))

5. // apt password = raw Base62 activation code (not a hash — must be verifiable by apt-auth.py)
   write /etc/apt/auth.conf.d/rusnas.conf via cockpit.file superuser:
     machine activate.rusnas.ru/apt/
     login  <serial>
     password <raw_base62>

6. render license status UI from payload
```

**Operator public key** embedded as a PEM string constant in `license.js` at build time (copied from `rusnas-license-server/operator_public.pem` by `build-deb.sh`).

## 11. Developer Workflow (for future Claude Code sessions)

When implementing a new feature:

| Action | What to do |
|--------|-----------|
| New Cockpit page | Add `page.html` + `js/page.js` + **manual entry in `manifest.json`** |
| New Python backend script | Add `js/scripts/newscript.py` — compiled to .pyc automatically |
| New system dependency | Add to `DEBIAN/control` Depends field |
| Security patch release | `./release.sh security X.Y.Z` |
| Feature release | `./release.sh main X.Y.Z` |
| New license feature | Add to `ALL_FEATURES` in `license-server/admin.py` |

**`manifest.json` is NOT auto-updated** — it must be edited manually when adding a new Cockpit page. The build glob picks up `*.html` files, but Cockpit will not show a page that has no manifest entry.

**No manual changes to `build-deb.sh`** unless the project structure changes fundamentally.
